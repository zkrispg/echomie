## 1. 测试目的与范围

### 1.1 测试目的

验证 EchoMie 在网页端（及与后端联调场景下）的核心能力是否可用：用户认证、媒体上传与异步处理、情绪卡片展示、周报、AI 对话、音乐列表、海报生成等。

### 1.2 测试范围

| 包含 | 不包含（或需单独环境） |
|------|-------------------------|
| Web 前端主要页面与交互 | 微信小程序真机全量回归（需备案域名 + HTTPS） |
| 后端 REST API 行为 | 第三方 DashScope / 微信接口 SLA 承诺 |
| Docker 部署形态下的服务联通 | 压力测试与性能基线 |

---

## 2. 测试环境

| 项 | 说明 |
|----|------|
| 客户端 | Chrome / Edge 最新稳定版（建议） |
| 访问地址 | 部署服务器公网 IP 或已绑定域名（示例：`http://<服务器IP>`） |
| 后端 | FastAPI + Gunicorn，PostgreSQL + Redis，Worker 队列 |
| 关键环境变量 | `DASHSCOPE_API_KEY`（大模型）、`JWT_SECRET_KEY`、`DATABASE_URL`、`REDIS_URL`、`STORAGE_BASE_PATH` 等 |
| 网络 | 需能访问 DashScope；若使用微信登录需配置 `WX_APP_ID` / `WX_APP_SECRET` |

### 5.2 建议

1. **部署后回归**：每次 `docker compose build` 后，至少执行 U-02、R-01、T-02、C-01 作为冒烟测试。  
2. **密钥与费用**：定期检查 DashScope 用量；生产环境勿将 `.env` 提交版本库。  
3. **小程序**：正式环境需配置合法 request 域名（HTTPS + 备案），与网页端分开验收。  
4. **备份**：生产数据库与 `STORAGE_BASE_PATH` 挂载卷建议制定备份策略。
